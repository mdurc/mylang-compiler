    - * yes we do...
      * decoupling global variable allocation from execution phase is important.
      * IR variables are now marked with an is_global flag based on the scope_id of zero.
      * before, it was an issue when the backend would route all top-level statements to the .bss section, even any supplemental (intermediate) IR variables used in a global initialization would be leaked into global memory. This prevents safe use of function calls and returning value-type structs at the global scope.
      * Now, the supplemental IR variables are used within the _start stack frame
      * structs returned by value during top-level initialization now allocate themselves to the hidden return pointer on the _start stack before being copied to memory.
      * backend is no longer reliant on tracking top-level assignment operations.
