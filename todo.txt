    - * yes we do...
      * decoupling global variable allocation from execution phase is important.
      * IR variables are now marked with an is_global flag based on the scope_id of zero.
      * before, it was an issue when the backend would route all top-level statements to the .bss section, even any supplemental (intermediate) IR variables used in a global initialization would be leaked into global memory. This prevents safe use of function calls and returning value-type structs at the global scope.
      * Now, the supplemental IR variables are used within the _start stack frame
      * structs returned by value during top-level initialization now allocate themselves to the hidden return pointer on the _start stack before being copied to memory.
      * backend is no longer reliant on tracking top-level assignment operations.

Writing custom assembler for the x86_64 subset we use -- eliminate nasm dependency, creating object files directly
* While writing this, we should try to keep in mind that we may add more backends in the future (ARM), so we should try to make it as agnostic as possible, though I'm not sure how much can be shared between backends.
- preprocessing:
  - [ ] somehow parse the NASM macros we use in the runtime asm library
  - [ ] tokenize the assembly that could be used from the backend implementation and the runtime asm library
- instruction encoding:
  - [ ] build opcode lookup table for the entire subset of instructions that could be generated by our compiler.
  - [ ] implement REX prefix emission logic for instructions using 64-bit operands and extended registers
  - [ ] implement ModR/M byte encoding for register to register and basic memory addressing
  - [ ] Implement Scale-Index-Base (SIB) byte encoding to support the array indexing [rdi + rax] (in the runtime asm)
- two-pass resolution:
  - [ ] find the exact byte length of every encoded instruction to build a symbol table of all labels and relative offsets
  - [ ] emit machine code, resolving local relative jumps by using the computed offsets (PIC)
- object file:
  - [ ] define the target object headers (probably just distinguishing between elf64 and macho64)
  - [ ] pack the encoded bytes into the text segment, string literals into the data segment, etc
  - [ ] generate relocation records for external system calls and global variables so the linker can patch them
  - [ ] verify the output byte-for-byte against a nasm generated object file
